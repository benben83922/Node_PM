# 本地端個人專案管理系統｜系統總覽

本文件定義以 **Obsidian + OpenClaw** 為核心的本地端個人 PM 系統架構，涵蓋系統定位、核心問題、解決方案、模組組成與實作路線圖。

---

## 第一部分：系統定位與核心問題

### 1. 使用者輪廓

| 項目 | 描述 |
| :--- | :--- |
| **角色** | 同時管理多個軟體專案的 PM |
| **主要工具** | Claude Code（文件產出）、GitHub（版控協作）、Notion（現有儀表板） |
| **核心痛點** | 文件散落、進度不透明、需要切換多個工具才能掌握全局 |
| **目標** | 用最少工具成本，建立一個「本地優先、即時同步、可用自然語言詢問」的 PM 系統 |

### 2. 核心問題

#### 問題一：文件產出與可視化之間有落差

PM 用 Claude Code 產出大量高品質的 `.md` 文件（PRD、ERD、Architecture...），但這些文件只是**靜態文字**。缺乏一個能把文件**自動轉化為可視化儀表板**的機制，導致：

- 每次確認進度都要手動開文件翻找
- 多個專案並行時，全局視角只存在於腦袋中
- 工程師更新 GitHub 後，PM 端沒有即時反映

#### 問題二：詢問進度需要切換工具

目前 PM 需在 Notion、GitHub、Claude Code 之間來回切換才能回答：「A 專案的 API 規格完成了嗎？B 專案目前在哪個 phase？」等問題。缺乏一個**自然語言查詢介面**，讓 PM 能直接用口語提問。

#### 問題三：文件沒有統一的機器可讀格式

文件缺乏統一的結構化 metadata，導致任何自動化工具（無論是 Dataview 或 AI）都無法跨文件做系統性查詢。

---

## 第二部分：解決方案三大支柱

### 支柱一｜文件即資料（Document-as-Data）

所有 Claude Code 產出的 `.md` 文件加入**統一 YAML Frontmatter**，讓文件同時具備「人可讀的內容」與「機器可讀的 metadata」。

```
文件本身的文字內容
    +
YAML Frontmatter（project / doc_type / status / phase / priority...）
    ↓
同一份文件，雙重用途
```

### 支柱二｜Obsidian 作為可視化儀表板

將所有 `.md` 文件同步到 **Obsidian Vault**，透過 **Dataview Plugin** 的 SQL-like 查詢，從 frontmatter 動態渲染：

- 多專案狀態總覽表
- 各 phase 進度追蹤
- 風險與待決事項清單
- 文件關聯圖（Graph View）

文件中的 **Mermaid 圖表**（ERD、架構圖、狀態機）由 Obsidian 原生渲染，零額外設定。

### 支柱三｜OpenClaw 作為自然語言查詢層

**OpenClaw** 連接 Obsidian Vault 目錄作為知識庫，讓 PM 能透過任何慣用通訊頻道（Telegram、Slack 等）直接用中文提問：

- 「A 專案的 ERD 完成了嗎？」
- 「目前哪些文件還在草稿狀態？」
- 「B 專案最高風險是什麼？」
- 「哪些 open questions 還沒決定？」

## 第三部分：系統架構總覽

```
┌─────────────────────────────────────────────────────────────────┐
│                      文件產出層                                   │
│                                                                  │
│   Claude Code  →  產出 .md（含 YAML Frontmatter）                │
│         ↓                                                        │
│   git push → GitHub Repository                                   │
└─────────────────────────────┬───────────────────────────────────┘
                              │  自動同步（Cron / Webhook）
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      本地儲存層                                   │
│                                                                  │
│   ~/ObsidianVault/_Projects/                                     │
│       ├── ProjectA/   ← git clone 或 symlink                    │
│       ├── ProjectB/                                              │
│       └── ProjectC/                                              │
└─────────────┬───────────────────────────────┬───────────────────┘
              │                               │
              ↓                               ↓
┌─────────────────────────┐   ┌───────────────────────────────────┐
│    可視化儀表板層         │   │      自然語言查詢層               │
│                         │   │                                   │
│   Obsidian              │   │   OpenClaw                        │
│   ├── Dataview          │   │   ├── 讀取 Vault 目錄             │
│   │   (進度表/風險表)    │   │   ├── 理解 Frontmatter 結構       │
│   ├── Kanban            │   │   └── 透過 Telegram/Slack 回答    │
│   ├── Graph View        │   │                                   │
│   └── Mermaid 渲染      │   │                                   │
└─────────────────────────┘   └───────────────────────────────────┘
```

## 第四部分：功能模組概覽

### 模組 A｜YAML Frontmatter 規範

定義所有 `.md` 文件必須包含的 metadata 欄位，是整個系統的資料地基。

| 欄位 | 說明 | 範例值 |
| :--- | :--- | :--- |
| `project` | 專案名稱 | `"ProjectA"` |
| `doc_type` | 文件類型 | `PRD / ERD / Architecture / WBS / API` |
| `status` | 文件狀態 | `draft / in-review / approved` |
| `phase` | 專案階段 | `planning / dev / testing / done` |
| `priority` | 優先度 | `low / medium / high / critical` |
| `owner` | 負責人 | `PM / TL / BE / FE` |
| `updated` | 最後更新日期 | `2026-04-25` |
| `tags` | 彈性標籤 | `[auth, backend]` |

> 完整規格：詳見 `/核心功能/文件規範_YAML設計規格書.md`

---

### 模組 B｜GitHub → Obsidian 自動同步

確保 PM 端的 Obsidian 永遠與 GitHub 保持一致，不需手動拉取。

| 方案 | 觸發方式 | 延遲 | 複雜度 |
| :--- | :--- | :--- | :--- |
| Cron Pull（推薦） | 每 1 分鐘 | ≤1 分鐘 | 低 |
| GitHub Actions Webhook | git push 後即時 | <1 分鐘 | 中 |

> 完整規格：詳見 `/核心功能/GitHub同步系統設計規格書.md`

---

### 模組 C｜Obsidian 儀表板

提供多種視圖，讓 PM 從不同角度掌握全局。

| 視圖類型 | 實現方式 | 用途 |
| :--- | :--- | :--- |
| 多專案狀態總覽 | Dataview TABLE | 一眼看所有專案文件狀態 |
| 進行中事項 | Dataview LIST | 過濾 phase = dev 的文件 |
| 風險清單 | Dataview TABLE | 過濾 priority = critical |
| 最近更新 | Dataview TABLE | 7 天內更新的文件 |
| 架構/ERD 圖 | Mermaid（原生） | 直接從 .md 渲染 |
| 任務看板 | Kanban Plugin | WBS 任務拖拉管理 |
| 文件關聯圖 | Graph View（原生） | 節點圖視覺化關係 |

> 完整規格：詳見 `/核心功能/Obsidian儀表板設計規格書.md`

---

### 模組 D｜OpenClaw 自然語言查詢

讓 PM 能用口語中文提問，OpenClaw 讀取 Vault 文件後回答。

| 問題類型 | 範例 | OpenClaw 查找目標 |
| :--- | :--- | :--- |
| 專案進度 | 「A 專案現在到哪裡了？」 | 該專案所有文件的 `phase` 欄位 |
| 文件狀態 | 「哪些文件還在草稿？」 | `status = draft` |
| 風險查詢 | 「目前最高風險是什麼？」 | WBS/PRD 的風險章節 |
| 技術細節 | 「B 專案的 ERD 有哪些主要實體？」 | ERD.md 內容 |
| 待決事項 | 「哪些問題還沒決定？」 | PRD Open Questions 章節 |

> 完整規格：詳見 `/核心功能/OpenClaw自然語言查詢設計規格書.md`

## 第五部分：與現有工具的關係

| 工具 | 現有用途 | 系統整合後的角色 | 是否可取代 |
| :--- | :--- | :--- | :--- |
| **Claude Code** | 產出 .md 文件 | 維持不變，加入 frontmatter 規範 | 不取代 |
| **GitHub** | 版控與工程師協作 | 維持不變，新增同步腳本 | 不取代 |
| **Notion** | 現有 PM 儀表板 | 可並行，待 Obsidian 穩定後逐步遷移 | 長期可取代 |
| **Obsidian** | 新增 | 本地可視化儀表板 | 取代 Notion |
| **OpenClaw** | 新增 | 自然語言查詢層 | 新增能力 |

## 第六部分：實作路線圖

| 階段 | 時程 | 交付項目 | 驗收標準 |
| :--- | :--- | :--- | :--- |
| **Phase 0：規範建立** | Day 1 | 更新 CLAUDE.md，固定 YAML frontmatter 格式 | 新產出文件自動帶入 frontmatter |
| **Phase 1：同步建立** | Day 1-2 | 建立 Obsidian Vault 目錄結構 + sync 腳本 | git push 後 1 分鐘內 Vault 自動更新 |
| **Phase 2：儀表板** | Day 3 | 安裝 Plugins，建立 `_Index.md` 主儀表板 | Dataview 查詢正確渲染 |
| **Phase 3：NL 查詢** | Day 4-5 | OpenClaw 連接 Vault，測試自然語言查詢 | 能正確回答「X 專案目前進度？」 |
| **Phase 4：微調** | Day 6-7 | 優化 Dataview 查詢 + OpenClaw system prompt | PM 日常使用流暢 |

## 第七部分：成功指標

| 指標 | 目標值 |
| :--- | :--- |
| **同步延遲** | git push 後 ≤ 1 分鐘 Obsidian 反映 |
| **查詢準確率** | OpenClaw 回答正確率 > 90% |
| **工具切換次數** | PM 掌握全局所需的工具數 ≤ 2（Obsidian + OpenClaw） |
| **文件覆蓋率** | 所有 Claude Code 產出的 .md 均含 frontmatter ≥ 100% |
