# 本地端個人專案管理系統｜系統總覽

本文件定義本地端個人 PM 系統架構，涵蓋系統定位、核心問題、解決方案、模組組成與實作路線圖。系統由七個功能模組組成（A：YAML 規範、B：GitHub 同步、C：Obsidian 知識庫、D：OpenClaw 查詢、E：WBS 進度控管、F：GitHub Actions 管道、G：Web App 儀表板）。

---

## 第一部分：系統定位與核心問題

### 1. 使用者輪廓

| 項目 | 描述 |
| :--- | :--- |
| **角色** | 同時管理多個軟體專案的 PM |
| **主要工具** | Claude Code（文件產出）、GitHub（版控協作）、Notion（現有儀表板） |
| **核心痛點** | 文件散落、進度不透明、需要切換多個工具才能掌握全局 |
| **目標** | 用最少工具成本，建立一個「本地優先、即時同步、可用自然語言詢問」的 PM 系統 |

### 2. 核心問題

#### 問題一：文件產出與可視化之間有落差

PM 用 Claude Code 產出大量高品質的 `.md` 文件（PRD、ERD、Architecture...），但這些文件只是**靜態文字**。缺乏一個能把文件**自動轉化為可視化儀表板**的機制，導致：

- 每次確認進度都要手動開文件翻找
- 多個專案並行時，全局視角只存在於腦袋中
- 工程師更新 GitHub 後，PM 端沒有即時反映

#### 問題二：詢問進度需要切換工具

目前 PM 需在 Notion、GitHub、Claude Code 之間來回切換才能回答：「A 專案的 API 規格完成了嗎？B 專案目前在哪個 phase？」等問題。缺乏一個**自然語言查詢介面**，讓 PM 能直接用口語提問。

#### 問題三：文件沒有統一的機器可讀格式

文件缺乏統一的結構化 metadata，導致任何自動化工具（無論是 Dataview 或 AI）都無法跨文件做系統性查詢。

---

## 第二部分：解決方案三大支柱

### 支柱一｜文件即資料（Document-as-Data）

所有 Claude Code 產出的 `.md` 文件加入**統一 YAML Frontmatter**，讓文件同時具備「人可讀的內容」與「機器可讀的 metadata」。

```
文件本身的文字內容
    +
YAML Frontmatter（project / doc_type / status / phase / priority...）
    ↓
同一份文件，雙重用途
```

### 支柱二｜雙路視覺化（Obsidian 知識庫 + Web App 儀表板）

**Obsidian**：將所有 `.md` 文件同步到本地 Vault，提供 PM 個人知識庫功能：

- 文件關聯圖（Graph View）
- 架構/ERD 圖（Mermaid 原生渲染，零額外設定）
- 各專案文件全文閱讀與個別儀表板（`_Dashboard.md`）

**Web App**：透過 GitHub Actions 將 WBS 任務與里程碑寫入 Supabase，再由 React Web App 渲染三角色儀表板：

- PM：多專案健康度總覽、S-Curve 診斷、Overdue 任務清單
- 工程師：個人待辦、Kanban 看板
- 客戶：交付摘要、里程碑時間軸

### 支柱三｜OpenClaw 作為自然語言查詢層

**OpenClaw** 連接 Obsidian Vault 目錄作為知識庫，讓 PM 能透過任何慣用通訊頻道（Telegram、Slack 等）直接用中文提問：

- 「A 專案的 ERD 完成了嗎？」
- 「目前哪些文件還在草稿狀態？」
- 「B 專案最高風險是什麼？」
- 「哪些 open questions 還沒決定？」

## 第四部分：功能模組概覽

### 模組 A｜YAML Frontmatter 規範

定義所有 `.md` 文件必須包含的 metadata 欄位，是整個系統的資料地基。

| 欄位 | 說明 | 範例值 |
| :--- | :--- | :--- |
| `project` | 專案名稱 | `"ProjectA"` |
| `doc_type` | 文件類型 | `PRD / ERD / Architecture / WBS / API` |
| `status` | 文件狀態 | `draft / in-review / approved` |
| `phase` | 專案階段 | `planning / dev / testing / done` |
| `priority` | 優先度 | `low / medium / high / critical` |
| `owner` | 負責人 | `PM / TL / BE / FE` |
| `updated` | 最後更新日期 | `2026-04-25` |
| `tags` | 彈性標籤 | `[auth, backend]` |

> 完整規格：詳見 `/核心功能/文件規範_YAML設計規格書.md`

---

### 模組 B｜GitHub → Obsidian 自動同步

確保 PM 端的 Obsidian 永遠與 GitHub 保持一致，不需手動拉取。

| 方案 | 觸發方式 | 延遲 | 複雜度 | 適用對象 |
| :--- | :--- | :--- | :--- | :--- |
| Obsidian Git 外掛（採用） | 每 1 分鐘（Obsidian 開啟時） | ≤1 分鐘 | 低 | 全體成員 |
| GitHub Actions Webhook | git push 後即時 | <1 分鐘 | 高 | - |

> 完整規格：詳見 `/核心功能/GitHub同步系統設計規格書.md`

---

### 模組 C｜Obsidian 個人知識庫

Obsidian 定位為 PM 的**個人知識庫**，提供本地文件的閱讀與知識連結。進度儀表板功能已移至 Web App。

| 功能 | 實現方式 | 用途 |
| :--- | :--- | :--- |
| 架構/ERD 圖渲染 | Mermaid（原生） | 直接從 .md 渲染圖表 |
| 文件關聯圖 | Graph View（原生） | 節點圖視覺化文件關係 |
| 文件全文閱讀 | Obsidian 閱讀模式 | 含 Frontmatter + 正文的完整知識庫 |

> 完整規格：詳見 `/核心功能/Obsidian儀表板設計規格書.md`

---

### 模組 D｜OpenClaw 自然語言查詢

讓 PM 能用口語中文提問，OpenClaw 讀取 Vault 文件後回答。

| 問題類型 | 範例 | OpenClaw 查找目標 |
| :--- | :--- | :--- |
| 專案進度 | 「A 專案現在到哪裡了？」 | 該專案所有文件的 `phase` 欄位 |
| 文件狀態 | 「哪些文件還在草稿？」 | `status = draft` |
| 風險查詢 | 「目前最高風險是什麼？」 | WBS/PRD 的風險章節 |
| 技術細節 | 「B 專案的 ERD 有哪些主要實體？」 | ERD.md 內容 |
| 待決事項 | 「哪些問題還沒決定？」 | PRD Open Questions 章節 |

> 完整規格：詳見 `/核心功能/OpenClaw自然語言查詢設計規格書.md`

---

### 模組 E｜WBS 模組進度控管

從文件層級向下展開至任務層級，以 **Markdown Task（`- [ ]`）格式**記錄子任務，搭配 GitHub Actions 解析後寫入 Supabase，讓 Web App 能呈現任務完成率。

| 功能 | 實現方式 | 用途 |
| :--- | :--- | :--- |
| 子任務格式 | `- [ ]` / `- [x]` + `[owner:: 角色:姓名]` | 追蹤至子任務層級 |
| 任務解析 | GitHub Actions Python 腳本 | 寫入 Supabase `tasks_sync` |
| 看板操作 | Obsidian Kanban Plugin（`Kanban.md`） | 拖拉式管理介面 |

> 完整規格：詳見 `/核心功能/WBS設計規格書.md`

---

### 模組 F｜GitHub Actions 資料同步管道

偵測 `_Projects/` 下的 `.md` 變更，解析結構化資料並寫入 Supabase。

| 觸發條件 | 解析目標 | 寫入目標 |
| :--- | :--- | :--- |
| push 到 main，`_Projects/**/*.md` 有變更 | YAML Frontmatter、WBS 任務、里程碑表格 | Supabase `tasks_sync`、`milestones` |

> 完整規格：詳見 `/核心功能/GitHub_Actions_Pipeline設計規格書.md`

---

### 模組 G｜Web App 團隊進度儀表板

讀取 Supabase 資料，依角色呈現三層鑽取式儀表板。

| 角色 | L1 | L2 | L3 |
| :--- | :--- | :--- | :--- |
| **PM** | 專案組合總覽 | 診斷中心（S-Curve / CFD） | 任務執行明細 |
| **工程師** | 今日待辦 | 技術上下文（Kanban） | 任務詳情 |
| **客戶** | 交付摘要 | 功能路徑圖（Roadmap） | 透明化文件 |

> 完整規格：詳見 `/核心功能/Web_App設計規格書.md`

## 第五部分：與現有工具的關係

| 工具 | 現有用途 | 系統整合後的角色 | 是否可取代 |
| :--- | :--- | :--- | :--- |
| **Claude Code** | 產出 .md 文件 | 維持不變，加入 frontmatter 規範 | 不取代 |
| **GitHub** | 版控與工程師協作 | 維持不變，新增 Actions Workflow | 不取代 |
| **Notion** | 現有 PM 儀表板 | 由 Web App 取代 | 可取代 |
| **Obsidian** | 新增 | PM 個人知識庫（文件閱讀、Mermaid、Graph View） | 不取代 Notion，角色不同 |
| **OpenClaw** | 新增 | PM 個人自然語言查詢層 | 新增能力 |
| **Supabase** | 新增 | 結構化進度資料庫（GitHub Actions 寫入、Web App 讀取） | 新增能力 |
| **Web App（React）** | 新增 | 團隊共用進度儀表板（取代 Notion） | 取代 Notion |

## 第六部分：實作路線圖

| 階段 | 時程 | 交付項目 | 驗收標準 |
| :--- | :--- | :--- | :--- |
| **Phase 0：規範建立** | Day 1 | 更新 CLAUDE.md，固定 YAML frontmatter 格式與 WBS 里程碑格式 | 新產出文件自動帶入 frontmatter |
| **Phase 1：同步建立** | Day 1-2 | 建立 Obsidian Vault 目錄結構 + 安裝 Obsidian Git 外掛（全體成員） | git push 後 1 分鐘內 Vault 自動更新 |
| **Phase 2：知識庫** | Day 3 | 安裝 Templater Plugin，確認 Mermaid 與 Graph View 正常運作 | Obsidian 作為個人知識庫可正常使用 |
| **Phase 3：NL 查詢** | Day 4-5 | OpenClaw 連接 Vault，測試自然語言查詢 | 能正確回答「X 專案目前進度？」 |
| **Phase 4：資料管道** | Day 6-7 | 建立 GitHub Actions Workflow + Python 腳本，設定 Supabase Schema | push 後 ≤ 2 分鐘 Supabase 資料更新 |
| **Phase 5：Web App** | Day 8-14 | 建立 React Web App，接入 Supabase，實作三角色 Dashboard | 三角色可正常登入並查看各自視圖 |
