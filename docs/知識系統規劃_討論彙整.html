系統總覽

GitHub → Obsidian → 檢索層 → OpenClaw → Discord

你已經完成了 AI 知識系統最難的一塊：把 GitHub 上的專案知識，自動同步成 AI 可存取的知識庫。

GitHub（唯一真實來源 Single Source of Truth）
   │  每個專案的 .md 文件
   ▼
Obsidian Git 外掛（稀疏檢出 + 每分鐘自動 pull）
   ▼
obsidian_vault/  ←─ 本地知識庫資料夾
   ▼
Docker：OpenClaw（掛載 vault，可讀取所有 Markdown）
   ▼
Discord Bot（內建串接）
   ▼
你在 Discord 提問 → OpenClaw 搜尋 vault → 整理答案 → 回覆

這套系統的本質

它不是傳統 RAG。目前沒有向量資料庫、沒有 Embedding、沒有 Chunk 預處理——OpenClaw 直接把 Vault 當知識庫，用「檔案系統搜尋 + LLM 讀檔理解」來回答。

四個核心優點

單一真實來源GitHub 是唯一寫入源，不會有「Vault 更新了、索引沒更新」的同步問題
成本趨近於零不需要 Pinecone / Weaviate / Embedding API
版本可追蹤誰改的、何時改、改了什麼，GitHub 全留底
Obsidian 當介面Backlinks、Graph、Tags、Dataview 全保留
關鍵心法：Obsidian 只是「人類編輯與檢視」的介面，GitHub 才是知識來源。AI 的檢索鏈不該綁在某一台 Obsidian 裝置上。

5GB Markdown 的挑戰

純檔案搜尋在「文件量小、命名清楚」時很夠用；但你的規模已經是 5GB 純 Markdown，這就跨過了「先不要做向量化」的門檻。

核心事實：5GB Markdown ≈ 數億字元 ≈ 數千萬～上億 Token。任何 LLM 都不可能一次掃完整個 Vault。所以一定要有「先檢索、再把候選文件交給模型」的前置階段。

真正的瓶頸：不是模型，是「有沒有找到對的檔」

就算用最新的 Claude Sonnet，問題也分三層：

層次說明是不是瓶頸
模型理解力看到 JWT 就懂登入機制——Sonnet 幾乎不會錯不是
文件有沒有被找到文件寫「JWT」、你問「登入」，關鍵字搜尋可能整份漏掉常常是
Context Window5GB 遠超任何 context，必須先篩選；篩錯 = 高品質地胡說八道是

兩種查詢，解法完全不同（別混為一談）

查詢類型例子最佳解
語意找單篇「登入機制怎麼運作？」（文件沒寫「登入」二字）向量／全文檢索
跨專案聚合「哪些專案用 OAuth？」「近三個月提到 K8s 的會議」結構化 metadata 查詢（SQL）
常見誤區：用「哪些專案用 OAuth」當作要上向量的理由是錯的。聚合型查詢靠向量只會回 top-K 相似片段，不保證「全部」；這種「列出所有符合的」本質是 SELECT … WHERE，該交給結構化 metadata（也就是你已規劃的 Supabase 那條線）。

OpenClaw 能連 Supabase / 資料庫嗎？

能。經網路查證，現在的 OpenClaw 原生支援 MCP，這把前面的保留全部解開了。

結論：OpenClaw 可以直接連 Postgres / Supabase。 它支援 MCP（Model Context Protocol，stdio + HTTP/SSE 兩種傳輸），有 200+ 現成 MCP server，其中就包含 Postgres、GitHub、Notion、Slack 等。只要在 openclaw.json 加一段 mcpServers 設定即可。

設定長相（概念示意）

// openclaw.json
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres",
               "postgresql://USER:PASS@HOST:5432/db"]
    }
  }
}
// → 之後任何 agent 都能呼叫這個 server 暴露的查詢工具

那 NemoClaw 是什麼？

NemoClaw 是 NVIDIA 的開源 stack，把 OpenClaw 關進安全沙盒裡常駐執行：全本地推論、資料不離開裝置、內建 Discord/Telegram/Slack 等通訊頻道支援。換句話說——你現在的底座本身就是免費開源、且隱私友善的。

三種接法（依推薦度排序）

方案做法評價
A. OpenClaw 直連掛 Postgres MCP server，OpenClaw 自己查 DB / pgvector現可行 最乾淨
B. Bot 層檢索你自己的 main.py 先查向量庫拿 top-K，再把命中片段餵給 OpenClaw最可控 推薦
C. 自動同步本地庫Indexer 容器定時 pull GitHub → chunk+embedding → 本地向量庫5GB 適用

實務上 B + C 一起做 最穩：本地自動同步建索引，檢索卡在你能完全掌控的 Bot 層；「OpenClaw 直連」變成加分項而非卡關點。

完全免費的技術選型

自架路線下，整條鏈路都能用開源 + 本地達成，唯一真實成本是「一台常開的機器」（你自己的 PC/NAS 即可）。

環節免費方案說明
Agent 底座NemoClaw + OpenClaw開源、本地常駐、Discord 內建
推論模型本地 Nemotron / 你現有的 Claude 訂閱（經 Proxy）不另外付 API 費
Embedding本地模型（如 nomic-embed-text、bge-m3，經 Ollama）零 API 成本
向量庫pgvector / Qdrant / Chroma / LanceDB / sqlite-vec全部開源免費

向量庫怎麼挑（針對你的情境）

選項型態適合對非技術友善度
Chroma嵌入式（像 SQLite，免伺服器）快速起步、10萬～數百萬 chunk最高
LanceDB檔案型（磁碟索引）資料量大於記憶體高
sqlite-vec單一檔案極簡、可攜帶高
pgvectorPostgres 擴充已經在用 Postgres（你有 Supabase）中
Qdrant獨立服務速度優先、規模大中
對非技術夥伴的關鍵：選嵌入式 / 檔案型（Chroma、LanceDB、sqlite-vec）——它就是「一個檔案」，沒有要維運的資料庫伺服器，一個 docker compose up 全帶起來。
5GB 提醒：Supabase 免費方案只有 500MB，裝不下 5GB 的全文向量索引。所以「完全免費」必須走本地自架這條路；雲端免費額度只夠放「小而結構化的 PM metadata」。

兩份資料，兩個地方放

資料大小放哪
結構化 PM metadata（任務/里程碑，frontmatter 解析）幾 MBSupabase（小、給 Web App，OpenClaw 也能 MCP 查）
5GB 全文 + 向量索引embedding 後可能 10–20GB本地向量庫，別丟雲端免費層

非技術夥伴怎麼辦？

你卡住的點其實是個誤解：以為「每個夥伴都要在自己電腦架資料庫」。用 Discord 當入口，正解是集中式單一後端。

一句話方案：只有你一個人（技術方）跑「一台」後端，把 NemoClaw + 向量庫 + Bot 全包進一個 docker compose up。所有非技術夥伴什麼都不裝，只在 Discord 打字。資料庫是「一份、集中」，不是「每台一份」。

唯一的真實成本：一台「常開」的機器（電費）。不需要雲端付費、不需要每位夥伴安裝任何東西。若你已有一台 24h 開機的 PC 或 NAS，等於完全免費。

自架 Server vs 免自架

「免自架」技術上可行（用 Discord HTTP Interactions 走 serverless），但有一個無法迴避的取捨。

取捨三角：免自架 server ＋ 完全免費 ＋ 5GB —— 三者最多選二。「免費」這一角會在 5GB 卡住：託管向量庫的免費額度多半裝不下，免費 LLM API 也有速率上限。

兩條路的整體比較

面向自架 Server 本次選定免自架（Serverless）
常開機器需要一台（PC/NAS）不需要
費用免費（只花電費）5GB 下每月幾~幾十美元
可用 Claude 訂閱可（本地 Proxy）不可，須改免費層/付費 API
用得到 OpenClaw/NemoClaw是否，改純 serverless RAG
資料隱私全本地、不外流資料進雲端託管
維運心力機器要顧（關機就斷線）幾乎免維運
非技術夥伴安裝零零

為什麼本次選「自架」：你要求「完全免費」＋「資料 5GB」＋「保留 OpenClaw 與 Claude 訂閱」——這個組合只有自架能同時滿足。代價就是需要一台常開機器。
未來若想免自架：先把「純 .md」量小（去掉附件），若能塞進向量庫免費層，就能切換成「免費＋免自架」全 serverless；否則接受每月一點點向量庫費用。

建議架構 · 下一步 自架 Server 路線

把上面所有結論收斂成一張圖，加上可以馬上動手的步驟。

GitHub（5GB md，唯一來源）
   │
   ├─（已規劃）GitHub Actions ─解析 frontmatter/WBS─→ Supabase（小·結構化）──→ Web App
   │
   └─（新增）Indexer 容器 ─chunk + 本地 embedding─→ 本地向量庫 ┐
                                                              │
Discord → 你的 Bot(main.py) ─① 查本地向量庫拿 top-K ─────────┘
                            └─② 把命中片段交給 OpenClaw → 回答

  ↑ 以上全部包在「一台常開機器」的單一 docker compose 內

立刻可做（低成本、效益最大）

  1. 重新量「純 .md」大小，確認 5GB 是文字還是含附件（圖片/PDF/.git）。決定索引規模。
  2. 把 GitHub Actions → Supabase 的 Indexer 真的接起來。你的設計文件裡它還是「規劃中、未實作」。這條一通，跨專案聚合查詢就解決一大半，零向量成本。
  3. 強化檔名與目錄結構（例：ProjectA/auth/oauth.md）。命名清楚時，OpenClaw 光看路徑就能精準挑檔。

等檔案搜尋真的不夠用，再做

  1. 起一個本地向量庫（建議 Chroma / LanceDB，嵌入式免維運），跟 NemoClaw 同一個 docker-compose。
  2. Indexer 加上 embedding：chunk（500–1000 token）+ 本地 embedding 模型，寫進向量庫，附上 project_id / path 等 metadata。
  3. 做 Hybrid Search：向量 + 全文(BM25) + metadata 過濾，召回更穩。
  4. （加分）OpenClaw 掛 MCP 直連 Postgres，讓它也能查結構化資料。

不建議

❌ 每台設備各建索引改成集中式單一後端
❌ 5GB 索引丟雲端免費層裝不下、要付費、資料外流
❌ 太早引入 Qdrant先用嵌入式向量庫，需要時再升級
❌ 讓非技術夥伴自架 DB他們只該用 Discord
總結（自架路線）：Obsidian 只當編輯器、GitHub 當唯一來源、索引集中維護一份，全部包進一台常開機器的單一 docker compose。先驗證 5GB、先把 Supabase Indexer 接起來解決聚合查詢、向量化延後且用本地嵌入式向量庫、後端集中一台讓非技術夥伴零安裝。
