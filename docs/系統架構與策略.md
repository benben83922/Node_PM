# 本地端個人 PM 系統｜架構策略與執行藍圖

---

## I. 執行摘要（Executive Summary）

### A. 核心問題與機會

PM 在管理多個並行軟體專案時，面臨的核心障礙並非能力問題，而是**工具碎片化**導致的認知負擔。文件在 Claude Code 產出、透過 GitHub 傳遞後，PM 端沒有任何自動化的「接收與整理」機制，導致每次確認進度都依賴記憶或手動翻找。

這個問題的市場缺口是：**現有 PM 工具（Notion、Jira）都假設用戶願意在工具內直接輸入資料**，而不是讀取外部 `.md` 文件並自動建立視圖。

### B. 我們的策略性答案

以**本地檔案系統為唯一資料源（Single Source of Truth）**，讓 Obsidian 作為渲染層、OpenClaw 作為查詢層，建立一個「零資料重複輸入」的 PM 工作流程。

### C. 三大設計原則

1. **本地優先（Local-First）**：所有資料存在本地，不依賴任何雲端訂閱服務的特定功能
2. **文件即資料（Docs-as-Data）**：YAML Frontmatter 讓 `.md` 同時服務人類閱讀與機器查詢
3. **最小工具數（Minimal Tooling）**：整個系統僅增加 Obsidian + OpenClaw，不引入新的資料庫或後端

## II. 使用者分析：PM 的日常工作流程

### A. 現有工作流程（有問題的）

```
Claude Code 產出 .md
      ↓
git push → GitHub
      ↓
工程師讀取開發              ← 工程師的起點
      ↓
PM 需要掌握進度時：
  → 打開 Notion 手動更新    ← 重複輸入
  → 打開 GitHub 翻 commit   ← 沒有全局視角
  → 打開 Claude Code 翻文件  ← 沒有動態渲染
  → 用記憶回答利害關係人問題  ← 風險最高
```

**核心問題**：從文件產出到進度可視化之間存在**人工橋接**的缺口。

### B. 目標工作流程（改善後）

```
Claude Code 產出 .md（含 YAML Frontmatter）
      ↓
git push → GitHub
      ↓
自動同步（1 分鐘內）
      ↓
Obsidian Vault（自動更新視圖）
      ↓
PM 詢問進度時：
  → 打開 Obsidian 看儀表板  ← 一個工具，全局視角
  → 用 OpenClaw 自然語言提問 ← 不需要切換工具
```

### C. 使用者需求分類

| 需求類型 | 具體場景 | 現有解法 | 目標解法 |
| :--- | :--- | :--- | :--- |
| **全局視角** | 同時管理 3+ 個專案 | 在 Notion 手動維護 | Dataview 自動渲染 |
| **即時查詢** | 利害關係人臨時詢問進度 | 翻文件找答案 | OpenClaw 口語提問 |
| **圖表可視化** | 確認架構/ERD 設計 | 開文件看 Mermaid 原始碼 | Obsidian 直接渲染 |
| **風險追蹤** | 掌握高風險事項 | 記憶或 Notion 頁面 | Dataview 過濾 priority |
| **文件同步** | 工程師更新後 PM 同步 | 手動通知 | Cron 自動 pull |

## III. 競爭格局：現有 PM 工具的根本限制

### A. 工具比較

| 工具 | 優點 | 對 PM 的根本限制 |
| :--- | :--- | :--- |
| **Notion** | UI 美觀、協作方便 | 資料必須「住在 Notion 內」，無法自動讀取外部 `.md` |
| **Jira** | 功能完整、Scrum 支援好 | 重量級工具，需維護額外的 issue 系統 |
| **Linear** | 現代 UI、工程師友善 | 同 Jira，與 `.md` 文件系統無法直接整合 |
| **Obsidian（純筆記）** | 原生 `.md`、本地儲存 | 沒有自動同步，需手動更新 |
| **GitHub Projects** | 與 repo 整合 | 設計給工程師，非 PM 全局視角 |

### B. 我們的差異化定位

傳統 PM 工具的競爭策略是「功能完整性與協作」；本系統的策略是**「零重複輸入的文件優先 PM 工作流」**。

```
現有工具的假設：「用戶在工具內輸入資料」
本系統的假設：「文件是唯一資料源，工具只負責渲染」
```

這個根本假設的不同，使得本系統對「以 Claude Code 驅動文件產出的 PM」有天然的適配性。

## IV. 系統設計策略

### A. 資料架構策略：以 Frontmatter 為中心的查詢模型

系統的核心設計決策是**讓 YAML Frontmatter 承載所有狀態資訊**，而非在 `.md` 文件內容中塞入結構化資料。

```
每份 .md 文件的資訊分層：

Layer 1: YAML Frontmatter（機器讀）
  project / doc_type / status / phase / priority / owner / updated / tags

Layer 2: 文件標題 & 章節結構（人 + 機器讀）
  # 標題 / ## 章節

Layer 3: 文件正文（人讀為主）
  詳細內容、表格、Mermaid 圖
```

**決策理由**：Dataview Plugin 的查詢效率完全依賴 frontmatter，而正文內容由 OpenClaw 的 LLM 理解，兩者分工明確。

### B. 同步架構策略：Pull 優先於 Push

| 策略 | 描述 | 選擇理由 |
| :--- | :--- | :--- |
| **Cron Pull（採用）** | 本地定時拉取 GitHub | 實作簡單，本地端主控，無需設定 ngrok |
| GitHub Webhook Push | 雲端推送到本地 | 需要本地 server，增加複雜度 |
| Manual Pull | 手動執行 | 不自動化，不採用 |

Cron Pull 策略：每 1 分鐘執行一次 `git pull`，對所有 `~/ObsidianVault/_Projects/` 底下有 `.git` 的目錄操作。

### C. 查詢層策略：靜態視圖 + 動態問答

| 查詢模式 | 工具 | 適用場景 |
| :--- | :--- | :--- |
| **預設靜態視圖** | Obsidian Dataview | 每日打開看全局狀態，問題已知 |
| **即時動態問答** | OpenClaw | 問題不固定、需要深入文件內容、口語提問 |

兩者互補：Dataview 解決「已知問題的快速查看」，OpenClaw 解決「臨時問題的自然語言回答」。

## V. 關鍵技術決策記錄（ADR）

### ADR-001：Obsidian 作為儀表板層

**狀態**：已決定

**背景**：需要一個能讀取本地 `.md` 文件、渲染 Mermaid、支援動態查詢的可視化工具。

**決策**：選用 Obsidian + Dataview Plugin。

**理由**：
- 原生支援 `.md` 格式，零轉換成本
- Dataview 支援 SQL-like 查詢 frontmatter，功能剛好滿足需求
- Mermaid 圖表原生渲染，現有文件中的架構圖/ERD 直接可用
- 本地儲存，無訂閱費用，資料自主權完整

**取捨**：Obsidian 不支援多人即時協作（但這是個人使用場景，不需要）。

---

### ADR-002：OpenClaw 作為自然語言查詢層

**狀態**：已決定

**背景**：需要一個能讀取本地目錄 `.md` 文件、透過常用通訊頻道回答問題的 AI 助手。

**決策**：選用 OpenClaw，連接本地 Vault 目錄作為知識庫。

**理由**：
- 支援多種通訊頻道（Telegram、Slack 等），PM 可在慣用頻道提問
- 本地運行，不將文件內容上傳第三方雲端
- 支援自訂 System Prompt，可針對 PM 工作情境客製化

**待驗證**：
- [ ] OpenClaw 對中文 `.md` 的解析品質測試
- [ ] 大量文件時的查詢延遲測量

---

### ADR-003：YAML Frontmatter 欄位設計

**狀態**：已決定

**背景**：需要決定 frontmatter 的欄位組成，在「查詢彈性」與「維護成本」之間取得平衡。

**決策**：採用 7 個核心欄位 + 1 個可選欄位（tags）。

**理由**：
- `project`：跨文件分組的唯一依據
- `doc_type`：過濾文件類型的核心欄位
- `status`：文件生命週期狀態
- `phase`：專案執行階段
- `priority`：風險與優先度
- `owner`：責任歸屬
- `updated`：時間維度過濾
- `tags`：彈性擴充，不強制

**排除的欄位**：`deadline`（放在文件正文中，由 OpenClaw 讀取）、`milestone`（粒度太細，屬於 WBS 內容）。

## VI. 風險評估

### A. 關鍵風險矩陣

| 風險 ID | 類別 | 描述 | 可能性 | 影響 | 緩解策略 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **R-01** | **規範** | **Frontmatter 欄位不一致，導致 Dataview 查詢破掉** | 高 | 高 | 在 CLAUDE.md 固定格式，使用 Templater 自動套入 |
| **R-02** | **同步** | Cron 腳本靜默失敗，PM 看到過期資料 | 中 | 中 | 在 sync 腳本加入 log 檔，定期確認 |
| **R-03** | **查詢** | OpenClaw 對中文的理解品質不穩定 | 中 | 中 | 建立測試問題集，定期驗證準確率 |
| **R-04** | **擴展** | 專案數量增多導致 Dataview 渲染變慢 | 低 | 低 | 設計 per-project 儀表板，避免單一全局查詢 |

### B. 重點關注：Frontmatter 一致性是系統地基

**整個系統能否正常運作，最關鍵的單一依賴是 Frontmatter 格式的一致性。** Dataview 的查詢、OpenClaw 的結構化理解，都建立在每份文件的 frontmatter 欄位名稱與值域一致的基礎上。

**首要任務**：在啟動其他模組之前，必須先在 `CLAUDE.md` 中固定 frontmatter 格式，並測試 Claude Code 能否穩定帶入正確欄位。

## VII. 執行藍圖

### A. 各 Phase 詳細計畫

#### Phase 0：規範建立（Day 1）

- **目標**：建立文件規範，讓所有新產出文件自動帶入 frontmatter
- **工作項目**：
  1. 更新 `CLAUDE.md`，加入 frontmatter 規範與範例
  2. 測試 Claude Code 產出文件是否正確帶入 frontmatter
  3. 手動補上現有文件的 frontmatter（一次性工作）
- **驗收**：產出 3 份含正確 frontmatter 的 `.md` 文件

#### Phase 1：同步建立（Day 1-2）

- **目標**：建立 GitHub → Obsidian 自動同步機制
- **工作項目**：
  1. 建立 `~/ObsidianVault/_Projects/` 目錄結構
  2. 將現有 GitHub repos clone 到對應目錄
  3. 建立 `~/scripts/sync_vault.sh` 腳本
  4. 設定 crontab（每 1 分鐘）
  5. 測試 push → 1 分鐘後 Obsidian 確認更新
- **驗收**：git push 後，Obsidian Vault 在 1 分鐘內自動更新

#### Phase 2：儀表板建立（Day 3）

- **目標**：建立 Obsidian 主儀表板與各類視圖
- **工作項目**：
  1. 安裝 Dataview、Kanban、Templater、Git Plugins
  2. 建立 `_Index.md`（多專案總覽儀表板）
  3. 建立 `_Risk_Board.md`（風險與待決事項看板）
  4. 確認 Mermaid 圖表正確渲染
- **驗收**：Dataview 能正確渲染多專案狀態表

#### Phase 3：自然語言查詢（Day 4-5）

- **目標**：OpenClaw 連接 Vault，能回答 PM 的進度問題
- **工作項目**：
  1. 設定 OpenClaw 讀取 `~/ObsidianVault/_Projects/` 目錄
  2. 撰寫並套入 PM 專用 System Prompt
  3. 測試 10 個標準問題，確認回答準確率
- **驗收**：10 個測試問題中，答對 9 個以上

#### Phase 4：微調（Day 6-7）

- **目標**：根據實際使用調整 Dataview 查詢與 OpenClaw 回答品質
- **工作項目**：
  1. 調整 frontmatter 欄位（若有遺漏場景）
  2. 新增 per-project 儀表板（若需要）
  3. 優化 OpenClaw System Prompt
- **驗收**：PM 能在無需說明的情況下，流暢使用系統一天

### B. 關鍵指標（KPIs）

| 指標 | 目標值 | 量測方式 |
| :--- | :--- | :--- |
| **同步延遲** | ≤ 1 分鐘 | git push 後計時至 Obsidian 反映 |
| **Dataview 查詢準確率** | 100%（正確渲染） | 手動驗證每個查詢 |
| **OpenClaw 回答準確率** | ≥ 90% | 10 題測試集 |
| **每日工具切換次數** | ≤ 2（Obsidian + OpenClaw） | 主觀評估一週 |
| **文件 Frontmatter 覆蓋率** | 100% | 統計有無 frontmatter 的文件比例 |
