# 本地端個人 PM 系統｜架構策略與執行藍圖

---

## I. 執行摘要（Executive Summary）

### A. 核心問題與機會

PM 在管理多個並行軟體專案時，面臨的核心障礙並非能力問題，而是**工具碎片化**導致的認知負擔。文件在 Claude Code 產出、透過 GitHub 傳遞後，PM 端沒有任何自動化的「接收與整理」機制，導致每次確認進度都依賴記憶或手動翻找。

這個問題的市場缺口是：**現有 PM 工具（Notion、Jira）都假設用戶願意在工具內直接輸入資料**，而不是讀取外部 `.md` 文件並自動建立視圖。

### B. 我們的策略性答案

以 **`.md` 文件為唯一寫入源（Single Source of Truth）**，讓 Obsidian 作為個人知識庫、NemoClaw + Discord Agent 作為查詢層、Supabase + Web App 作為雲端多角色儀表板，建立一個「零資料重複輸入」的 PM 工作流程。

### C. 三大設計原則

1. **文件即資料（Docs-as-Data）**：YAML Frontmatter 讓 `.md` 同時服務人類閱讀與機器查詢
2. **單向寫入（Unidirectional Write）**：所有寫入只透過 Claude Code → git push，視圖工具（Obsidian、Web App）只讀
3. **按場景選擇工具**：本地個人閱讀 → Obsidian + NemoClaw；雲端多角色共用 → GitHub Actions + Supabase + Web App

## II. 使用者分析：PM 的日常工作流程

### A. 現有工作流程（有問題的）

```
Claude Code 產出 .md
      ↓
git push → GitHub
      ↓
工程師讀取開發              ← 工程師的起點
      ↓
PM 需要掌握進度時：
  → 打開 Notion 手動更新    ← 重複輸入
  → 打開 GitHub 翻 commit   ← 沒有全局視角
  → 打開 Claude Code 翻文件  ← 沒有動態渲染
  → 用記憶回答利害關係人問題  ← 風險最高
```

**核心問題**：從文件產出到進度可視化之間存在**人工橋接**的缺口。

### B. 目標工作流程（改善後）

```
Claude Code / 工程師 產出 .md（含 YAML Frontmatter）
      ↓
git push → GitHub
      ↓
路徑 A：Obsidian Git Plugin（≤1 分鐘）→ Obsidian Vault（個人知識庫）
路徑 B：GitHub Actions（≤2 分鐘）→ Supabase → Web App（三角色儀表板）
      ↓
PM 詢問進度時：
  → 打開 Web App 看儀表板  ← 多角色共用全局視角
  → 用 NemoClaw（Discord）自然語言提問 ← 深度文件查詢
  → 打開 Obsidian 查閱文件  ← 個人知識庫
工程師：Web App 查看個人待辦 / Kanban
客戶：Web App 查看里程碑 / 交付進度
```

### C. 使用者需求分類

| 需求類型 | 具體場景 | 現有解法 | 目標解法 |
| :--- | :--- | :--- | :--- |
| **全局視角** | 同時管理 3+ 個專案 | 在 Notion 手動維護 | Web App L1 自動渲染（資料來自 Supabase） |
| **即時查詢** | 利害關係人臨時詢問進度 | 翻文件找答案 | NemoClaw Discord 提問 |
| **圖表可視化** | 確認架構/ERD 設計 | 開文件看 Mermaid 原始碼 | Obsidian 直接渲染 |
| **風險追蹤** | 掌握高風險事項 | 記憶或 Notion 頁面 | Web App L2 診斷（S-Curve、CFD、Overdue） |
| **本地文件同步** | 工程師更新後 PM 同步 | 手動通知 | Obsidian Git Plugin 自動 pull |
| **多角色協作** | 工程師/客戶查看進度 | 需要 PM 手動匯出報告 | Web App 三角色各自登入查看 |

## III. 競爭格局：現有 PM 工具的根本限制

### A. 工具比較

| 工具 | 優點 | 對 PM 的根本限制 |
| :--- | :--- | :--- |
| **Notion** | UI 美觀、協作方便 | 資料必須「住在 Notion 內」，無法自動讀取外部 `.md` |
| **Jira** | 功能完整、Scrum 支援好 | 重量級工具，需維護額外的 issue 系統 |
| **Linear** | 現代 UI、工程師友善 | 同 Jira，與 `.md` 文件系統無法直接整合 |
| **Obsidian（純筆記）** | 原生 `.md`、本地儲存 | 沒有自動同步，需手動更新 |
| **GitHub Projects** | 與 repo 整合 | 設計給工程師，非 PM 全局視角 |

### B. 我們的差異化定位

傳統 PM 工具的競爭策略是「功能完整性與協作」；本系統的策略是**「零重複輸入的文件優先 PM 工作流」**。

```
現有工具的假設：「用戶在工具內輸入資料」
本系統的假設：「文件是唯一資料源，工具只負責渲染」
```

這個根本假設的不同，使得本系統對「以 Claude Code 驅動文件產出的 PM」有天然的適配性。

## IV. 系統設計策略

### A. 資料架構策略：以 Frontmatter 為中心的查詢模型

系統的核心設計決策是**讓 YAML Frontmatter 承載所有狀態資訊**，而非在 `.md` 文件內容中塞入結構化資料。

```
每份 .md 文件的資訊分層：

Layer 1: YAML Frontmatter（機器讀）
  project / doc_type / status / phase / priority / owner / updated / tags

Layer 2: 文件標題 & 章節結構（人 + 機器讀）
  # 標題 / ## 章節

Layer 3: 文件正文（人讀為主）
  詳細內容、表格、Mermaid 圖
```

**決策理由**：Dataview Plugin 的查詢效率完全依賴 frontmatter，而正文內容由 NemoClaw 的 LLM 理解，兩者分工明確。

### B. 同步架構策略：雙路衍生

| 路徑 | 機制 | 目的地 | 延遲 | 用途 |
| :--- | :--- | :--- | :--- | :--- |
| **路徑 A：本地知識庫** | Obsidian Git Plugin auto pull | Obsidian Vault | ≤1 分鐘 | PM 個人閱讀、NemoClaw 查詢 |
| **路徑 B：雲端資料庫** | GitHub Actions → Python 腳本 | Supabase | ≤2 分鐘 | Web App 三角色儀表板 |

兩條路徑都從同一個 git push 觸發，服務不同場景，互不干擾。

### C. 查詢層策略：分層查詢

| 查詢模式 | 工具 | 適用場景 |
| :--- | :--- | :--- |
| **多角色結構化視圖** | Web App（Supabase → React） | 每日全局狀態、多角色協作、S-Curve/CFD |
| **個人本地靜態視圖** | Obsidian Dataview（個別儀表板） | PM 個人 per-project 文件狀態查閱 |
| **即時自然語言問答** | NemoClaw（索引整個 Vault，Discord 頻道提問） | 臨時問題、深入文件內容、口語提問 |

三層查詢互補：Web App 解決「多角色共用的結構化追蹤」，Obsidian 解決「個人深度閱讀」，NemoClaw 解決「臨時自然語言查詢」。

## V. 關鍵技術決策記錄（ADR）

### ADR-001：Obsidian 作為個人知識庫層

**狀態**：已決定

**背景**：需要一個能讀取本地 `.md` 文件、渲染 Mermaid、支援動態查詢的可視化工具。

**決策**：選用 Obsidian + Dataview Plugin。

**理由**：
- 原生支援 `.md` 格式，零轉換成本
- Dataview 支援 SQL-like 查詢 frontmatter，功能剛好滿足需求
- Mermaid 圖表原生渲染，現有文件中的架構圖/ERD 直接可用
- 本地儲存，無訂閱費用，資料自主權完整

**取捨**：Obsidian 不支援多人即時協作（但這是個人使用場景，不需要）。

---

### ADR-002：NemoClaw + Discord Agent 作為自然語言查詢層

**狀態**：已決定

**背景**：需要一個能讀取本地目錄 `.md` 文件、透過常用通訊頻道回答問題的 AI 助手。

**決策**：採用 NemoClaw（Docker 沙盒中運行 OpenClaw）+ Discord Bot，PM 在 Discord 頻道提問，NemoClaw 讀取整個 Vault 後回答。

**理由**：
- Discord 為工程師團隊慣用通訊頻道，無需額外切換工具
- Docker 沙盒隔離 OpenClaw 運行環境，避免污染本地環境
- 本地運行，不將文件內容上傳第三方雲端
- 支援自訂 System Prompt，可針對 PM 工作情境客製化

**待驗證**：
- [ ] NemoClaw 對中文 `.md` 的解析品質測試
- [ ] 大量文件時的查詢延遲測量

---

### ADR-003：YAML Frontmatter 欄位設計

**狀態**：已決定

**背景**：需要決定 frontmatter 的欄位組成，在「查詢彈性」與「維護成本」之間取得平衡。

**決策**：採用 8 個核心欄位（`project / doc_type / status / phase / priority / owner / updated / tags`）+ WBS 專用欄位（total_tasks / module_count / team）。

**理由**：
- `project`：跨文件分組的唯一依據
- `doc_type`：過濾文件類型的核心欄位
- `status`：文件生命週期狀態
- `phase`（WBS）：整個專案的宏觀階段，由 GitHub Actions 同步至 Supabase `projects.current_phase`
- `priority`：風險與優先度
- `owner`：文件負責人（非任務負責人）
- `updated`：時間維度過濾
- `tags`（選填）：彈性標籤，供 Graph View 分群、NemoClaw 細粒度過濾

WBS 專用欄位（非全文件適用）：`team`（角色 → `{name, email}` 字典，供 GitHub Actions 解析 `assignee_email`）

**排除的欄位**：`progress`（Web App 動態計算，不靜態存）、`milestone`（屬於 WBS 正文表格內容）。

---

### ADR-004：Obsidian Git 外掛作為本地同步機制

**狀態**：已決定

**背景**：需要一個讓全體成員（包含非技術成員）都能自動同步 GitHub 文件的機制。

**決策**：採用 Obsidian Git 外掛，設定 Auto pull interval: 1 分鐘。

**理由**：無需 Terminal；跨平台；適合全體成員；push 前自動 merge 降低衝突風險。

**取捨**：Obsidian 需保持開啟才能持續同步，但開啟時會自動補齊離線期間的更新。

---

### ADR-005：Supabase 作為雲端進度資料庫

**狀態**：已決定

**背景**：需要一個能讓 Web App 讀取結構化進度資料的後端，支援 RBAC 與 Realtime。

**決策**：採用 Supabase（managed PostgreSQL），啟用 Row Level Security。

**理由**：零維護後端；內建 Auth + RLS；Realtime 訂閱；service_role key 讓 Actions 繞過 RLS 寫入，anon key 讓前端受控讀取。

**取捨**：引入雲端依賴；免費方案 500MB 對純文件 metadata 資料量足夠。

---

### ADR-006：React Web App 作為團隊儀表板

**狀態**：已決定

**背景**：需要一個多角色共用的進度儀表板，Obsidian 定位為 PM 個人知識庫，無法服務工程師和客戶。

**決策**：建立 React Web App，部署至 Vercel，讀取 Supabase 資料；PM/工程師提供 L1/L2/L3 鑽取式視圖，客戶（Viewer）提供 L1/L2 摘要視圖。

**理由**：彈性最高；Recharts 支援 S-Curve/CFD 圖表；Vercel 零配置部署；三角色視圖清晰分離。

---

### ADR-007：GitHub Actions 作為資料同步管道

**狀態**：已決定

**背景**：需要在不引入額外基礎設施的情況下，自動將 `.md` 文件的結構化資料寫入 Supabase。

**決策**：GitHub Actions Workflow 偵測 `**/*.md` 變更，執行 Python 腳本解析並 upsert Supabase。

**理由**：零額外基礎設施；與 git push 無縫整合；失敗時 GitHub 自動通知；支援手動重觸發。

**取捨**：強依賴 GitHub；解析 Markdown 的 regex 需謹慎維護。

## VI. 風險評估

### A. 關鍵風險矩陣

| 風險 ID | 類別 | 描述 | 可能性 | 影響 | 緩解策略 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **R-01** | **規範** | **Frontmatter 欄位不一致，導致 Dataview 查詢破掉或 Actions 解析失敗** | 高 | 高 | 在 CLAUDE.md 固定格式；使用 Templater 自動套入；Phase 0 驗收 3 份文件 |
| **R-02** | **同步** | Obsidian 未開啟時不自動同步，PM 看到過期本地文件 | 低 | 低 | 開啟時自動補齊；Web App 資料來自 Supabase，不受此影響 |
| **R-03** | **查詢** | NemoClaw 對中文的理解品質不穩定 | 中 | 中 | 建立 10 題測試問題集，Phase 3 驗收；不穩定時降級查 Obsidian |
| **R-04** | **管道** | GitHub Actions 腳本執行失敗，Supabase 資料未更新 | 低 | 中 | GitHub 自動發送 fail 通知；支援手動重觸發；`.md` 文件本身不受影響 |
| **R-05** | **安全** | Supabase RLS 政策設定錯誤，用戶可查看未授權專案 | 低 | 高 | Phase 4 以三角色測試驗收 RLS 政策正確性 |
| **R-06** | **擴展** | 專案數量增多導致 Dataview 全局查詢渲染變慢 | 低 | 低 | 超過 5 個專案後改為 per-project 儀表板；主要進度查看已移至 Web App |

### B. 重點關注：Frontmatter 一致性是系統地基

**整個系統能否正常運作，最關鍵的單一依賴是 Frontmatter 格式的一致性。** Dataview 的查詢、NemoClaw 的結構化理解，都建立在每份文件的 frontmatter 欄位名稱與值域一致的基礎上。

**首要任務**：在啟動其他模組之前，必須先在 `CLAUDE.md` 中固定 frontmatter 格式，並測試 Claude Code 能否穩定帶入正確欄位。

## VII. 執行藍圖

### A. 各 Phase 詳細計畫

#### Phase 0：規範建立（Day 1）

- **目標**：建立文件規範，讓所有新產出文件自動帶入 frontmatter
- **工作項目**：
  1. 更新 `CLAUDE.md`，加入 frontmatter 規範與範例
  2. 測試 Claude Code 產出文件是否正確帶入 frontmatter
  3. 手動補上現有文件的 frontmatter（一次性工作）
- **驗收**：產出 3 份含正確 frontmatter 的 `.md` 文件

#### Phase 1：同步建立（Day 1-2）

- **目標**：建立 GitHub → Obsidian 自動同步機制
- **工作項目**：
  1. 初始化 `~/ObsidianVault/` 父儲存庫，各專案以 git submodule 加入至 Vault 根目錄
  2. 開啟稀疏檢出，確保各 submodule 只同步 `.md` 文件
  3. 安裝並設定 Obsidian Git 外掛（Update submodules: 開啟、Auto pull interval: 1 分鐘、Pull on startup: 開啟）
  4. 各成員執行相同設定，確認全員 Vault 同步正常
  5. 測試 push → 1 分鐘後 Obsidian 確認更新
- **驗收**：git push 後，Obsidian Vault 在 1 分鐘內自動更新

#### Phase 2：Obsidian 知識庫（Day 3）

- **目標**：Obsidian 作為 PM 個人知識庫正常運作
- **工作項目**：
  1. 安裝 Dataview、Kanban、Templater、Obsidian Git Plugins
  2. 確認 Mermaid 圖表正確渲染、Graph View 正常顯示
  3. 為各專案建立 `_Dashboard.md`（per-project 本地視圖）
- **驗收**：Obsidian 作為個人知識庫可正常閱讀、查詢文件

#### Phase 3：自然語言查詢（Day 4-5）

- **目標**：NemoClaw + Discord Agent 連接整個 Vault，能回答 PM 的進度問題
- **工作項目**：
  1. 以 Docker 啟動 NemoClaw 沙盒，設定讀取 `~/ObsidianVault/` 目錄
  2. 設定 Discord Bot 連接 NemoClaw（`main.py` Flask + Discord bot）
  3. 撰寫並套入 PM 專用 System Prompt（含 `[owner::]` WBS 格式說明）
  4. 測試 10 個標準問題，確認回答準確率
- **驗收**：10 個測試問題中，答對 9 個以上

#### Phase 4：資料管道（Day 6-7）

- **目標**：建立 GitHub Actions → Supabase 資料同步管道
- **工作項目**：
  1. 在 Supabase 建立 Schema（5 張表 + RLS 政策）
  2. 設定 GitHub Actions Secrets（SUPABASE_URL / SUPABASE_KEY）
  3. 建立 Workflow + Python 腳本（`scripts/sync_to_supabase.py`）
  4. 測試 push 後 Supabase 資料正確更新
- **驗收**：push 後 ≤ 2 分鐘 Supabase tasks_sync 資料更新

#### Phase 5：Web App 儀表板（Day 8-14）

- **目標**：建立三角色 Web App，讓工程師和客戶也能查看進度
- **工作項目**：
  1. 建立 React Web App，接入 Supabase
  2. 實作三角色 Dashboard（PM / 工程師：L1/L2/L3 鑽取式；客戶：L1/L2 摘要視圖）
  3. 部署至 Vercel
  4. 驗收三角色 RBAC 正確運作
- **驗收**：三角色可正常登入並查看各自視圖；Supabase Realtime 即時反映任務更新

### B. 關鍵指標（KPIs）

| 指標 | 目標值 | 量測方式 |
| :--- | :--- | :--- |
| **本地同步延遲** | ≤ 1 分鐘 | git push 後計時至 Obsidian 反映 |
| **雲端同步延遲** | ≤ 2 分鐘 | git push 後計時至 Supabase tasks_sync 更新 |
| **Dataview 查詢準確率** | 100%（正確渲染） | 手動驗證每個查詢 |
| **NemoClaw 回答準確率** | ≥ 90% | 10 題測試集 |
| **每日工具切換次數** | ≤ 2（Web App + NemoClaw） | 主觀評估一週 |
| **文件 Frontmatter 覆蓋率** | 100% | 統計有無 frontmatter 的文件比例 |
| **RBAC 正確性** | 100% | 三角色各自測試存取範圍 |
