Notion 同步規劃
GitHub → Notion
單向鏡像 + 多人協作
討論彙整 · 2026-06-01

需求與結論

GitHub（事實來源） → Obsidian-Git → 各自本地 OpenClaw → 一個 Action → Notion（唯讀鏡像）

在既有的「GitHub → Obsidian → OpenClaw → Discord」知識系統之上，希望文件被修改時也能同步到 Notion，讓多人一起閱讀，且不走付費方案。

一句話結論：把 Notion 當成 GitHub 的單向唯讀鏡像（編輯一律走 OpenClaw / Obsidian），用一個中央 GitHub Action 在 push 時把 .md 推上 Notion。同步機制全程 0 元；只要 Notion 工作區維持單人 + 公開發佈給團隊讀，整套免費。

討論是怎麼一步步收斂的

  1. 提出需求：文件改動同步到 Notion，多人讀／編輯，免費。
  2. 點出核心矛盾：不可能在沒有「真雙向同步」的前提下，讓 GitHub 與 Notion 同時當寫入主控。先定事實來源。
  3. 一度選了雙向（C）：設計過「檔案級 last-write-wins + 狀態表 + 衝突備份」的方案，但坦白它是維護地獄。
  4. 關鍵轉折：改成「所有編輯都走 OpenClaw → GitHub → 單向推 Notion」，跨系統雙向衝突直接消失，退回最乾淨的 A 模式。
  5. 最終架構：GitHub 當事實來源、成員皆協作者、各自本地 OpenClaw、Notion 單向鏡像 —— 本質是標準 Git 多人協作。
  6. 確認費用：API / Actions / 轉換套件全免費；唯一付費陷阱是 Notion 自己的方案限制。
關鍵心法：「多人一起編輯」與「免費」會打架。把編輯收斂到單一路徑（OpenClaw/Obsidian），Notion 退成唯讀，才能同時免費又無衝突。

核心矛盾：誰是事實來源？

所有設計分歧都源自一個問題——文件的「編輯權威來源（single source of truth）」放在哪裡。

不可違反的原則：沒有「真雙向同步 + 衝突處理」的前提下，不可能讓兩個系統同時當寫入主控。若 GitHub 是事實來源，每次 push 就會用 GitHub 的內容覆蓋 Notion，團隊在 Notion 上的編輯會被沖掉。

原本的單向資料流

Obsidian / OpenClaw 編輯
   ▼
GitHub（事實來源）
   ▼
Obsidian 地端  +  Supabase（結構化 metadata）

要加 Notion，第一個決定不是「用什麼工具」，而是「編輯到底發生在哪裡」。這個決定直接決定方案從「半天搞定」到「長期維護地獄」。

別做的事：不要嘗試「block 級智慧合併」。當同一份檔案兩邊改了不同段落，想自動合併就是把自己推進坑裡。免費且可維護的雙向，最多只能做到檔案級 last-write-wins。

關鍵洞察：把寫入收斂成單一路徑

與其處理「GitHub ↔ Notion」雙向衝突，不如讓所有編輯只走一條路（OpenClaw / Obsidian → GitHub），Notion 只單向接收。如此一來跨系統衝突從根本上不存在，問題退化成大家熟悉的 Git 多人協作。

三種同步模式

依「編輯發生在哪裡」分成三條路。最終選 A。

模式編輯發生在難度 / 可靠度免費
A. Notion 唯讀鏡像 最終選定只在 Obsidian / OpenClaw；Notion 給團隊讀＋留言簡單、超穩✅
B. Notion 當編輯權威只在 Notion → 反向同步回 GitHub → Obsidian中等 要寫 Notion→md 轉換 + 定時抓取✅
C. 真雙向同步兩邊都能改很難、脆弱 衝突處理 + 語法失真✅ 但成本高

為什麼放棄 C（真雙向）

即便堅持要 C，免費可維護的做法也只能是「檔案級 last-write-wins + 狀態表 + 衝突備份 + Discord 告警」，不做段落合併：

  • 狀態表 .notion-sync/state.json 記錄每檔的 page_id / md_hash / notion_hash / 時間戳。
  • 只有「上次同步後兩邊都改過同一檔」才算衝突 → 依時間戳判贏家，輸的版本存成 .conflict.md 不刪除，發 Discord 告警人工處理。
  • 必做回音抑制：正向寫入 Notion 會更新其 last_edited_time，反向輪詢要靠 hash 比對跳過自己剛寫的，否則無限同步。
C 最痛的地方：雙向 round-trip 一定在 Dataview、[[wikilink]]、Mermaid、frontmatter 這些 Obsidian 專屬語法上掉資料；加上衝突邏輯 + 回音抑制同時 debug，很容易整套崩掉。結論：避開 C。

最終架構 A 模式 · 多人協作 + 單向鏡像

GitHub 當事實來源、團隊成員皆協作者、各自本地跑 OpenClaw、Notion 是雲端單向唯讀鏡像。本質就是「Git 多人協作 + 一條單向鏡像」。

                 GitHub（.md，唯一事實來源）
                 │
   ┌─────────────┼──────────────┬────────────────────┐
   │ Obsidian-Git│ Obsidian-Git │ Obsidian-Git        │ 一個中央 GitHub Action
   ▼             ▼              ▼                      ▼ (push 觸發，單向)
 成員A 本地     成員B 本地      成員C 本地             Notion（雲端，唯讀鏡像）
   │             │              │                      ▲
 本地 OpenClaw  本地 OpenClaw   本地 OpenClaw          團隊用瀏覽器讀 / 留言
可行，而且是最務實的一版。它把問題從「自創跨系統雙向同步」降回「用 Git 既有機制做多人協作 + 一條單向 Notion 鏡像」——後者是被解決幾十年的老問題。

資料流拆解

動作路徑
讀 / 問成員 → 各自本地 OpenClaw → 讀各自同步下來的 vault
改OpenClaw 改本地檔 → git pull --rebase → commit → push 回 GitHub

校正一個概念：Notion 是雲端服務，沒有「本地副本」。正確說法是「GitHub → Notion（雲端）單向鏡像」，團隊在瀏覽器讀。

這版解掉了什麼

✅ 跨系統雙向衝突消失Notion 唯讀，無 Notion→GitHub 回寫，反向同步整套不用做
✅ 退回標準 Git 協作衝突交給 git 的 merge / rebase，不必自創機制
✅ 語法失真不再雙向Dataview / wikilink 只正向輸出，沒人改回來

殘留風險與三道防護

衝突沒「消失」，是搬家到一個小很多、好解很多的地方：單機寫入競態與一般 Git 協作。以下都可解，但別忽略。

1. 一台機器上「兩個 git 駕駛」會打架 最易踩

本地同時有 Obsidian-Git 插件 與 OpenClaw 都可能碰 git，時序交錯就會亂。

防護：一台機器只留一個 git 駕駛。建議讓 OpenClaw 改完檔自己做 pull --rebase → commit → push，Obsidian-Git 只負責「拉下別人的更新」（設成只 pull / 關掉 auto-push）。

2. 兩人改同檔同段 → 真的會冒出衝突標記

這是此架構唯一會出現 <<<<<<< 的地方。更糟的是 Obsidian-Git 不會幫非技術成員漂亮解衝突，帶標記的壞 .md 會被同步到所有人、甚至推上 Notion 變垃圾。

防護：① push 前一律 git pull --rebase，失敗自動重試；② Notion 同步 Action 加防呆——偵測到含 <<<<<<< 衝突標記的檔就拒絕同步並發 Discord 告警；③ 縮短 Obsidian-Git auto-pull 間隔（1–2 分鐘）。

3. Notion 同步必須「集中跑一份」

GitHub→Notion 用一個中央 GitHub Action（push 觸發）即可。若讓每台本地機器也去推 Notion，就變成 N 個寫入者搶同一個 Notion → 又製造競態。

4. 最終一致性有延遲

A 改完 push 後，B 的本地要等 Obsidian-Git 下次 pull 才看得到，因此 B 可能在舊版上開始改而撞車。靠第 2 點的 pull-rebase + 短輪詢間隔吸收。

5. 稀疏檢出（sparse checkout）的範圍限制

若成員只稀疏檢出部分檔案，他的 OpenClaw 只能改「本地有的檔」；要改範圍外的得先擴大 checkout。是範圍限制，不是 bug。

另一個非衝突風險：內容保真度。自然語言改檔有不確定性——成員說「更新 X 段」，LLM 可能順手動到別段。建議 OpenClaw 改完把 git diff 回貼 Discord 確認，或讓改動走 PR / 至少留 commit 可回溯。

怎麼把 GitHub 同步到 Notion

機制就是一條 GitHub Action，掛在現有「push → 解析 → 寫 Supabase」那條 pipeline 旁邊。

git push（.md 變更）
   ▼
GitHub Action 觸發
   ▼
讀取變更的 .md → 用 martian 轉成 Notion blocks
   ▼
呼叫 Notion API，upsert 到 Notion 資料庫對應頁面

需要準備的東西

  • 一個 Notion integration token（在 notion.so/my-integrations 建立，免費），存進 GitHub Secret NOTION_TOKEN。
  • 把目標頁面 / 資料庫分享給這個 integration，並把 NOTION_DB_ID 也存進 Secret。

建議：用「一個 Notion Database」承載文件

把所有文件當成一個 Notion Database 裡的 pages，Database 的 properties 直接對應 YAML frontmatter：

Notion property對應 frontmatter型別
Title檔名title
project / doc_type / status / phase / priority / owner同名select / text
updatedupdateddate
_filepath / _reporepo 內路徑 / repo 名text
好處：frontmatter ↔ DB properties（零失真）、正文 ↔ page 內容 blocks，兩件事分開處理。團隊也直接得到一個能 filter / sort 的看板，等於把 Dataview 搬到 Notion 給多人用。多 repo 用 _repo property 區分。

免費技術選型

環節免費方案說明

提醒：是 Node 工具鏈，與現有的 Supabase Python 腳本各跑各的，互不干擾。

要付費嗎？

同步機制本身 100% 免費。唯一可能逼你付費的，是 Notion 自己的方案限制，跟同步技術無關。

後續更新：團隊已訂閱付費方案，本頁「單人 + 公開發佈」的免費權宜做法已不需要——可直接把團隊加成正式成員、無 block 上限、原生留言。流程也演進為「受控雙向」（非技術人員在 Notion 直接編輯）。詳見 →「流程演進：受控雙向」「三級總覽 · 定案」分頁。

⚠️ 唯一付費陷阱：Notion 的方案限制

Notion 免費方案，工作區只有 1 人 → 無限 blocks；一旦加入 2 人以上 → 整個工作區硬上限 1,000 blocks，超過後只能讀 / 編輯舊內容，無法再新增。你的知識庫是一堆規格書，幾份文件就會撞破 1,000。

怎麼維持「多人讀 + 全免費」

既然 Notion 已定為唯讀鏡像，正好有免費解：工作區只保留你一人（unlimited blocks），把資料庫頁面用「Publish to web / 發佈到網路」開公開連結，團隊用連結讀。
做法結果
單人工作區 + 公開發佈永遠免費、無限文件；讀者非成員，不觸發 1,000 block 上限；天生唯讀，吻合設計
把全團隊加成正式成員撞破 1,000 block → 被迫升 Plus（約 $8–10 / 人 / 月）
公開發佈的代價：拿到連結的人都能看（無細緻權限控管），且不能在上面留言 / 編輯。若要「成員身分、能在 Notion 內互動 / 留言」就得回到付費方案。
一句話：同步免費；只要 Notion 工作區維持單人 + 公開發佈給團隊讀，整套免費。要「多人是正式成員、能在 Notion 內互動」才需付費。

下一步

把結論收斂成可以動手的順序。先求穩，別一次把所有風險都打開。

落地步驟

  1. 建 Notion integration（免費）與一個 Database，properties 對齊 frontmatter；把 NOTION_TOKEN、NOTION_DB_ID 存進 GitHub Secret。
  2. 寫 scripts/sync_to_notion.mjs（正向），在現有 sync_to_supabase.yml 後加一個 step 或獨立 job。
  3. 把 Notion 工作區設為單人 + 公開發佈，產生團隊閱讀用的公開連結。
  4. 每台機器確立單一 git 駕駛：OpenClaw 自己 pull --rebase / commit / push，Obsidian-Git 只 pull。
  5. Action 加衝突標記防呆：含 <<<<<<< 的檔拒絕同步並發 Discord 告警。

建議的保守上線節奏

即使最終是多人協作，也先只跑正向同步（純鏡像）兩週，確認 markdown → Notion 的轉換品質與你的文件相容，再把多人寫入 / 防呆機制逐一打開。

不建議

❌ 做真雙向同步（C）免費但脆弱、語法失真、長期維護痛
❌ 讓人在 Notion 直接編輯會被下次 push 覆蓋；Notion 只能唯讀
❌ 每台本地各推 Notion變多寫入者競態，集中在一個 Action
❌ 全團隊加成 Notion 成員撞破 1,000 block 被迫付費；改用公開發佈
總結：把編輯收斂到 OpenClaw / Obsidian、GitHub 當唯一事實來源、Notion 用一個中央 Action 做單向唯讀鏡像、工作區單人 + 公開發佈給團隊讀。免費、無跨系統衝突，殘留的只是標準 Git 協作問題，用「單一 git 駕駛 + pull-rebase 重試 + Notion 衝突防呆」三道防護即可。

流程演進：受控雙向 需求變更

需求後來變了：要加入非技術人員，他們直接在 Notion 編輯，再叫 OpenClaw 同步回 GitHub；技術人員仍直接改本地 .md 上傳。於是系統從「單向鏡像」變成「雙向」。

誠實提醒：這正是前面刻意避開的真雙向同步（C 模式）。技術人員寫 GitHub→Notion、非技術人員寫 Notion→GitHub，只要同一份文件在上次同步後兩邊都被改過，就會互相覆蓋、掉資料。 
技術人員 → 改本地 .md → push GitHub ──→ (Action) ──→ Notion
非技術人員 → 在 Notion 改 ──→ (叫 OpenClaw) ──→ GitHub ←── 同一份檔！

讓它變安全的關鍵：一份文件只有一個「主人」

不要做智慧合併。改用 frontmatter 加 authority 欄位（或用資料夾切分），讓每份文件的權威來源只有一邊——系統層級「雙向」，但每份檔案都是單向，衝突機率趨近於零。

文件類型主人方向在另一邊
技術文件（規格、含 Dataview/WBS）GitHubGitHub→Notion 單向Notion 頁面鎖唯讀
內容文件（非技術人員寫的部分）NotionNotion→GitHub（叫 OpenClaw）技術人員不在 git 直接改

兩個方向怎麼做

方向一　GitHub → Notion不變：push → 中央 Action → REST API upsert。「GitHub 主人」的文件把 Notion 頁設成員唯讀。
方向二　Notion → GitHubDiscord 指令觸發 → 中央 OpenClaw 用 Notion API/MCP 讀頁 → notion-to-md → 由 DB properties 重建 frontmatter → 寫回 .md → pull-rebase → push。
修正先前建議：反向方向讓 OpenClaw 接 Notion MCP/API 是合理的，但只用來「讀」Notion；寫 GitHub 仍由 OpenClaw 用 git 完成，OpenClaw 絕不直接寫 Notion。

必備三道安全機制

機制作用
狀態表 .notion-sync/state.json每檔記 page_id / md_hash / notion_hash / 時間戳

WBS 任務狀態：會失真嗎？

非技術人員若要在 Notion 改 WBS 裡某個任務的狀態，會失真——但失真的不是勾選本身，是「把整頁轉回去」這個動作。

分兩段看

第一段　checkbox 本身：還好。- [ ] M3.1.3 … 對應 Notion to-do block，打勾 → 轉回就是 - [x]，兩邊都用勾選框表達同一件事，對應乾淨。
第二段　整頁 notion-to-md 轉回：災難。為了把勾選帶回 GitHub 而重新序列化整份 WBS，會把文件其他元素全丟進失真的轉換器。
WBS 內元素整頁轉回會發生什麼
[owner:: BE:張後端] 行內欄位很可能被跳脫成 \[owner:: …\] → Supabase 解析 regex 失敗
```dataview 區塊語言標記/內容掉失或被跳脫
## 里程碑 表格表格 round-trip 跑版
巢狀任務縮排（M3 ▸ M3.1 ▸ M3.1.3）縮排被壓平 / 層級改變
frontmatter / [[wikilink]] / Mermaid漂移或損壞
你以為的 diff 是「一個字元 [ ]→[x]」，實際 push 的卻是「整份檔案被重排」的大 diff。WBS 是最容易壞的那種文件。

正解：不要整頁轉回，做外科手術式改一行

反向同步（針對 WBS）：
  讀 Notion 的 to-do blocks
   → 從 block 文字抓出任務 ID（M3.1.3，穩定錨點）
   → 比對該 ID 在 WBS.md 那一行的勾選狀態
   → 只把該行的 [ ] ↔ [x] 翻轉，其他完全不碰
更推薦：把 WBS 任務做成 Notion Database。一列一任務（欄位：任務ID、狀態下拉、Owner、Deadline），非技術人員只改「狀態下拉」，不碰任何 markdown；反向同步讀每列狀態 → 用任務 ID 對回 WBS.md，只 patch 那一行。這跟既有的 tasks_sync 資料模型銜接自然，零失真。

環節為什麼 OK
GitHub 當唯一事實來源架構地基，版本可追溯
技術人員改 .md → push標準 git，本就為多人協作而生
Obsidian-Git pull 到各本地 vault單向拉取，無衝突
OpenClaw 讀本地 vault 回 Discord內容在本地、即時、無 API 限制
GitHub → Notion 單向同步（REST API）確定性腳本，免費，最穩
Notion 付費：成員讀 + 留言留言不動正文；付費已無 1,000 block 上限
frontmatter ↔ Notion DB properties結構化對應，零失真
任務狀態用「Notion DB 下拉」或「按任務 ID patch 單行」只動狀態，其餘 byte 不動，不失真
OpenClaw 用 Notion MCP/API 只讀（反向用）讀取沒問題；寫 GitHub 由 git 做

⚠️ 可行但有風險（要配防護）

環節風險防護
Notion → GitHub 反向同步（整個雙向）C 模式，同檔兩邊改會掉資料authority 一檔一主人，把系統雙向拆成每檔單向
同台機器兩個 git 駕駛時序交錯弄亂本地 git一台只一個駕駛：OpenClaw 自做 git，Obsidian-Git 只 pull
兩人改同檔同段產生 <<<<<<< 衝突標記push 前 pull --rebase + 重試；縮短 auto-pull 間隔
衝突標記檔被推上 Notion壞檔污染 NotionAction 偵測 <<<<<<< 拒推 + 告警
兩方向互相觸發無限回音迴圈狀態表存雙邊 hash，未變就跳過
非技術人員編輯內容型文件後反向notion-to-md 掉部分格式限定純文字/內容型；反向後回貼 diff 確認
反向同步由誰跑多人各自觸發 = 多寫入者競態集中在一個中央 OpenClaw 實例序列化
最終一致性延遲有人在舊版上改 → 撞車pull-rebase + 短輪詢吸收
OpenClaw 自然語言改檔LLM 可能動到非目標段（保真度）改完回貼 git diff 確認 / 走 PR

⛔ 紅線（會壞，別做）

別做原因
WBS / Dataview 結構化文件用整頁 notion-to-md 反向整份重序列化 → Dataview/表格/owner/縮排全壞。改狀態只能按任務 ID patch 單行
讓非技術人員在 Notion 直接改結構化文件正文同上，這類文件應 GitHub 單向 + Notion 鎖唯讀
讓 OpenClaw 直接「寫」Notion繞過 GitHub，重新製造雙向衝突，破壞事實來源
每台本地機器各自推 Notion多寫入者搶同一 Notion → 競態
做 block 級智慧合併雙向同步最大的坑，免費方案無法可靠維護
定案總結：綠燈＝GitHub 單向出去（→Notion、→各 vault、→OpenClaw 讀）整套 + 任務狀態用 DB 下拉/單行 patch，全穩。黃燈＝Notion→GitHub 反向整體可行，命脈是 authority 一檔一主人，再加狀態表/回音抑制/衝突備份/單一 git 駕駛兜底。紅燈＝結構化文件絕不走整頁反向、OpenClaw 絕不直接寫 Notion。
